WarehouseReceipt Token — Data Structure Design

Three approaches for the flexible metadata layer

Core Receipt (shared across all crops)

struct WarehouseReceipt {
    uint256       id;              // Unique token ID
    address       farmer;          // Asset Owner
    address       warehouseAgent;  // Verified Verifier
    address       mfi;             // Lender (MFI)
    uint256       quantityKg;      // Harvest volume (kg)
    uint256       expiryDate;      // Collateral expiry
    bytes32       warehouseId;     // Physical storage location
    ReceiptStatus status;          // Issued → Active → Claimed / Defaulted
    string        metadataUri;     // Crop-specific data (off-chain or on-chain)
}

Option A: Off-Chain Metadata (IPFS/URL)

Pros

  • Cheapest gas — tiny struct
  • Unlimited data per crop type
  • Easy to add new crops (off-chain schema)

Cons

  • MFI must fetch off-chain data
  • Data could be unavailable
  • Need external oracle for on-chain verification

Option B: On-Chain Mappings

// Crop registry — add new crops without touching core contract
mapping(bytes32 => CropDefinition) public cropDefinitions;

struct CropDefinition {
    string   name;
    string[] metricKeys;        // e.g. ["moisture_content", "grade", "defect_rate"]
    uint256  minAcceptableValue;
}

// Per-token metrics stored on-chain
mapping(uint256 => mapping(bytes32 => uint256)) public cropMetrics;

Pros

  • Everything on-chain — MFI reads directly
  • Crop definitions are contract-enforced
  • No external dependencies

Cons

  • Higher gas for deployment + each metric write
  • Rigid — adding crop types costs gas
  • Each metric is same type (uint256), limits flexibility

Option C: Hybrid — On-Chain Summary + Off-Chain Detail

// On-chain: just the essentials for the MFI to assess
struct CropSummary {
    bytes32 cropType;            // hash of "MAIZE"
    uint256 estimatedValueUsd;   // computed by warehouse agent
    uint256 qualityScore;        // 0-100 composite (grade + moisture)
    string  metadataUri;         // IPFS: full inspection report
}

Pros

  • Key value metrics on-chain for MFI
  • Full inspection report off-chain
  • Balances gas cost with verifiability

Cons

  • Two data sources to maintain
  • EstimatedValueUsd must be trusted (or oracle-backed)

Which metadata approach fits your tri-party model best?

A

Off-Chain Only (IPFS)

Minimal on-chain struct. MFI fetches details from IPFS. Requires off-chain trust.

B

Full On-Chain Mappings

Everything on-chain. Crop registry contract for extensibility. Highest verifiability.

C

Hybrid (Recommended)

On-chain value summary + IPFS detail. Best balance for MFI lending decisions.
